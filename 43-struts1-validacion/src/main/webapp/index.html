43 — Struts 1.x: Dónde poner la validación

En Struts 1.x hay tres sitios posibles para la validación de un formulario. El módulo 40 mostró el primero. Este módulo muestra los otros dos.

Comparativa de enfoques

Módulo ¿Dónde vive la validación? Mecanismo Configurable sin recompilar
40 Action.execute() Código Java en la acción No
43-A validation.xml Struts Validator (declarativo) Sí — editar el XML
43-B ActionForm.validate() Código Java en el Form No
Clave común a 43-A y 43-B: en struts-config.xml el action lleva validate="true" y un atributo input="...". Struts llama a form.validate() antes de action.execute(); si hay errores, reenvía a input sin llamar nunca a la Action.

Flujo de una petición POST (ambos ejemplos)

Navegador POST /xxx.do │ ▼ ActionServlet ├─ 1. Crea/reutiliza el form bean y lo puebla por reflexión ├─ 2. validate="true" → llama a form.validate() │ ┌─ hay errores → reenvía a input="..." (Action NO se ejecuta) │ └─ sin errores → continúa └─ 3. Llama a Action.execute() ← solo si validate() devolvió vacío └─ mapping.findForward("exito") → vista de éxito

Ejemplos de este módulo

XML Ejemplo A — Struts Validator (declarativo)

Formulario de registro con tres campos: nombre, correo electrónico y edad.
Las reglas de validación se declaran en WEB-INF/validation.xml. RegistroForm extiende ValidatorForm y no contiene ningún método validate() propio: el framework lo hereda y ejecuta el XML. RegistroAction.execute() no tiene ni una línea de validación.

Reglas aplicadas:

  • nombre — obligatorio, mínimo 2 caracteres, máximo 50
  • email — obligatorio, formato de correo válido
  • edad — obligatorio, número entero, rango 1–120

Piezas clave:

  • RegistroForm extends ValidatorForm
  • WEB-INF/validation.xml con <field depends="required,email,intRange">
  • ValidatorPlugIn en struts-config.xml que carga los XMLs al arrancar
  • validate="true" + input="..." en el <action>

Java Ejemplo B — Validación programática en el Form

Formulario de contacto con tres campos: nombre, correo electrónico y mensaje.
ContactoForm extiende ActionForm y sobreescribe el método validate(ActionMapping, HttpServletRequest). Cuando Struts detecta validate="true", llama a ese método antes de la Action. La lógica de validación está en Java pero en el Form, no en la Action.

Reglas aplicadas:

  • nombre — obligatorio
  • email — obligatorio, debe contener @ y algo después
  • mensaje — obligatorio, mínimo 10 caracteres

Piezas clave:

  • ContactoForm extends ActionForm con validate() sobreescrito
  • validate() construye y devuelve un ActionErrors con claves de mensajes
  • No necesita ValidatorPlugIn ni validation.xml
  • validate="true" + input="..." en el <action> (igual que A)

UTF-8 en los mensajes de error

Por defecto Struts 1.x carga ApplicationResources.properties como ISO-8859-1, lo que corrompe los acentos. Este módulo incluye Utf8MessageResourcesFactory y Utf8MessageResources: dos clases que sobreescriben el método de carga para usar UTF-8. El .properties se escribe con caracteres normales; no se necesitan escapes \uXXXX. 

<!-- struts-config.xml -->
<message-resources parameter="ApplicationResources"
                   factory="com.cursosdedesarrollo.Utf8MessageResourcesFactory"/>
